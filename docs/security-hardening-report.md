# AuthAny 安全加固报告

## 概述

本文档记录 AuthAny 本轮安全加固中已确认的问题、已经落地的修复项，以及当前的验证结果。

## 已修复的致命与高危问题

### 1. 登录暴力破解防护不足

- 问题：
  - 登录接口只有限流，没有账户锁定机制。
  - 登录失败没有审计记录。
  - 基于 IP 的限流依赖框架默认 IP 解析，存在被绕过风险。
- 修复：
  - 增加按账户维度的锁定机制，15 分钟内连续失败 5 次后锁定 15 分钟。
  - 收紧登录接口的 IP 限流阈值。
  - 为登录成功和登录失败都补充审计事件。
  - 引入显式客户端 IP 提取逻辑，只在可信代理链下解析转发头。

### 2. 代理信任配置可能导致 IP 伪造

- 问题：
  - 之前使用布尔型代理信任配置，粒度过粗，不适合安全敏感的限流场景。
- 修复：
  - 移除 `AUTHANY_TRUST_PROXY`。
  - 新增 `AUTHANY_TRUSTED_PROXIES`。
  - 只有当直接连接对端属于显式配置的可信代理时，才信任 `X-Forwarded-For`。

### 3. 加密密钥回退复用了 Cookie 密钥

- 问题：
  - 机密加密曾允许回退到 `COOKIE_SECRET`。
- 修复：
  - 完全移除该回退逻辑。
  - `AUTHANY_APP_SECRET_ENCRYPTION_KEY` 现在在配置解析阶段就是必填项。

### 4. JWT 签名密钥可能在运行时静默生成

- 问题：
  - 非生产环境中，签名密钥曾允许在启动时自动生成。
- 修复：
  - 删除运行时生成签名密钥的逻辑。
  - JWT 私钥和公钥 PEM 现在必须显式提供。

### 5. 历史明文私钥回退路径可被接受

- 问题：
  - 签名密钥读取逻辑曾接受 metadata 中的 `private_key_pem` 明文字段。
- 修复：
  - 签名逻辑现在强制要求 `private_key_ciphertext`。
  - 后台密钥创建不再接受任意 metadata 注入。
  - 密钥创建被限制为生成式 `RS256` 密钥材料。

### 6. OAuth Token 端点缺少 IP 级限流

- 问题：
  - `/oauth/token` 之前只按客户端维度限流。
- 修复：
  - 为 `authorization_code`、`refresh_token`、`client_credentials` 三类交换统一补充 IP 级限流。

### 7. Refresh Token 重放后的处置范围过窄

- 问题：
  - 检测到 refresh token 重放时，之前只吊销单个客户端下的一组 token。
- 修复：
  - 现在会吊销该操作员在所有客户端下的全部活跃 refresh token。

### 8. 角色 Scope 可能导致权限提升

- 问题：
  - 角色带来的 scope 之前会与请求 scope 做并集，存在越权扩大权限的风险。
- 修复：
  - 有效 scope 现在被严格限制在客户端允许范围内。
  - `authany.admin` 只有在被显式请求、客户端允许、且操作员具备有效管理员角色时才会签发。

### 9. 跨租户的后台与委派查询约束不足

- 问题：
  - 多处后台和委派相关查询没有一致地按 `tenantId` 收敛。
- 修复：
  - 已为后台服务、审计查询、调用方凭证、委派主体解析、目标连接解析、当前操作员解析等链路补齐租户约束。

### 10. Hosted Login 与 Consent 表单缺少 CSRF 防护

- 问题：
  - HTML 表单 POST 之前没有 CSRF 保护。
- 修复：
  - 为托管登录表单和 OAuth 授权确认表单增加了签名 CSRF Token。

## 已修复的中危问题

### 11. 登录响应暴露内部操作员 ID

- 修复：
  - 已从 `/api/auth/login` 的成功响应中移除 `operator.id`。

### 12. 异常日志可能打印完整异常对象

- 修复：
  - HTTP 异常过滤器中不再直接打印原始异常对象，改为输出收敛后的异常信息。

### 13. 生产环境 Seed 默认值不安全

- 修复：
  - Seed 脚本现在在所有环境下都要求显式提供密码和密钥类输入。
  - 已彻底移除默认管理员密码、默认 OAuth Client Secret、默认调用方凭证。

### 14. JSON 哈希不稳定

- 修复：
  - 为委派上下文等结构化对象增加稳定序列化后再计算哈希。

### 15. Demo 与 Admin 应用存在不安全的密钥回退

- 修复：
  - 已移除 demo/admin 应用环境变量中的回退密钥。
  - 已移除 Demo UI 中暴露默认用户名密码的提示文本。

### 16. Session Cookie 架构风险已收口

- 问题：
  - 原先核心登录态是自签名 HMAC Cookie，不便于主动撤销，浏览器侧风险面较大。
- 修复：
  - 核心登录态已改为服务端持久会话。
  - 浏览器 Cookie 现在只保存随机会话 ID，不再保存可直接信任的会话载荷。
  - 当前用户恢复时会从 Redis 读取会话，再回查数据库确认操作员仍处于激活状态。

### 17. SecretEncryptionService 密钥派生方式已升级

- 问题：
  - 原先加密主密钥是对配置值直接做 `SHA-256` 派生。
- 修复：
  - 新密文格式已切换为基于 HKDF-SHA256 的密钥派生。
  - 派生上下文中加入用途和租户信息，避免简单直接哈希的弱约束问题。
  - 同时保留对历史 `v1` 密文的解密能力，用于平滑读取旧数据。

### 18. Redis 开发模式降级已移除

- 问题：
  - 之前 Redis 不可用时，非生产环境会静默回退到内存存储。
- 修复：
  - 已彻底移除内存降级。
  - 现在 Redis 不可用时，认证限流、会话、重放防护等依赖 Redis 的安全能力不会假装继续工作。
  - 健康检查也会准确反映 Redis 不可用状态。

## 验证情况

### 自动化验证

- `pnpm build`
- `pnpm test`

以上验证在本轮安全加固后均已通过。

### 依赖漏洞扫描

- 当前环境下 `pnpm audit` 无法完成，因为所配置的 registry 不提供 npm audit 所需的安全通告接口。
- `better-npm-audit` 也无法完成，因为当前工作区没有 `package-lock.json`。

### 攻击式验证

- 本轮曾使用临时攻击脚本验证以下场景：
  - 重复错误密码登录尝试
  - Token 端点的类暴力破解与 IP 轮换尝试
- 说明：
  - 在当前桌面环境里，临时拉起的隔离验证服务已经完成路由启动，但 HTTP 请求没有稳定返回，因此这里无法诚实地宣称“已完成最终实战攻击验证”。
  - 这些脚本没有作为长期维护入口保留在仓库中，后续验证应通过正式测试用例或独立审计工具执行。

## 本轮涉及的改动范围

- 认证与会话安全
- OIDC / OAuth 签发与校验
- 后台服务租户隔离
- 密钥与机密管理
- Seed 与运行时配置安全
- Demo / Admin 应用环境安全
- 安全验证脚本
