# 新项目通用安全检查清单

在项目上线前，以及每次涉及认证、授权、后台管理、系统集成的重大改动前，都应按这份清单逐项检查。

使用建议：

- 不要只检查主服务代码，还要一起检查：
  - 管理后台
  - Demo / 示例应用
  - seed 脚本
  - 攻击验证脚本
  - 健康检查与运维脚本
- 对每一项都要追问一句：
  - 缺失配置时会不会继续运行
  - 依赖故障时会不会静默降级
  - 校验缺失时会不会默认放行
- 如果答案是“会”，默认按高风险处理。

## 1. 密钥与机密管理

- 所有会承载真实流量的环境都必须显式配置机密值。
- 不要让一个密钥跨多个信任边界复用。
- 不允许加密主密钥回退到 Cookie / Session 密钥。
- JWT 签名密钥必须显式配置，或改为 KMS 托管。
- 不要在运行时静默生成签名密钥；如果系统支持后台创建密钥，也必须显式创建后才能签发 Token。
- 私钥必须加密存储。
- 禁止保留明文私钥 metadata 的兼容路径。
- 记录密钥 ID、启用时间、轮换时间。
- 历史验签公钥只能在受控的 Token 生命周期窗口内保留可用。
- `.env.example`、README、部署文档中不得保留已废弃的私钥 / 公钥环境变量名。
- 示例配置中不得出现看似可用的弱密钥占位值，并让人误以为可以直接上线。

## 2. 认证安全

- 登录成功和登录失败都必须审计。
- 对重复失败登录增加账户锁定或二次验证机制。
- 登录接口必须同时按客户端 IP 和账户标识限流。
- 不要信任 `X-Forwarded-For`，除非直接上游代理被明确列入可信名单。
- 从 Cookie 恢复登录态时，必须重新校验账户是否仍然有效。
- 非必要情况下，不要在登录响应里暴露内部用户 ID。
- 登录页面不得暴露默认账户、默认密码、演示凭据提示。
- 认证失败要区分：
  - 真正的凭据无效
  - 基础设施异常
  - 密钥加载失败
  - 数据库故障
- 不要把内部异常统一伪装成“用户名密码错误”或“JWT 无效”。

## 3. 会话安全

- 浏览器会话必须使用 `httpOnly` Cookie。
- 除本地开发外，应优先启用 `secure` Cookie。
- 所有会改变状态的浏览器表单 POST 都应有 CSRF 防护。
- `sameSite` 策略必须显式设置并定期复核。
- 当账户被禁用、冻结、删除后，既有会话必须失效。
- 会话恢复时必须回查持久化状态，不能只信任 Cookie 内载荷。
- 会话系统缺依赖时必须失败关闭，不能静默改用内存或无状态简化模式。

## 4. OAuth / OIDC 安全

- 每个客户端都必须校验允许的 grant type。
- 每个客户端都必须校验允许的 scope。
- Token 中签发的 scope 绝不能超出客户端允许范围。
- 适用授权码模式时，必须强制 PKCE。
- Redirect URI 必须精确匹配。
- Token 端点必须有 IP 级限流。
- 客户端级限流只能作为第二层，不能替代 IP 级限流。
- Refresh Token 一旦检测到重放，应足够广范围吊销。
- Discovery 和 JWKS 端点应设置合理缓存头。
- `requester token`、`target token`、`introspection` 这类接口在内部异常时不能伪装成普通认证失败。
- Token 签发面与状态面必须一致：
  - 只要 Token 已签发，就必须能在状态记录 / 审计里找到它
  - 不能出现“Token 发出去了，但状态系统不知道”的情况

## 5. 授权与后台接口

- 所有查询和写操作都必须按租户或等价安全边界收敛。
- 后台权限不能只信任 Token Claims，必要时应回查持久化权限状态。
- 审计、指标、密钥管理、机密查看等接口必须置于强管理员授权之后。
- 管理员专属 scope 必须同时满足“被请求”和“被策略允许”。
- Access Grant、Target Connection、Role Binding 等授权对象创建时，不要提供“allow / active / required”等危险默认值。
- 管理后台表单枚举值应显式下拉选择，不要靠自由文本输入安全关键字段。
- 后台详情页展示敏感配置时，要用人能看懂的字段视图，不要直接丢原始 metadata JSON 让操作员猜。

## 6. 存储与数据处理

- 对结构化对象做哈希前，应先做稳定序列化或规范化。
- 从数据库读取的未知 JSON 必须先校验再使用。
- 运行日志要避免泄露堆栈、SQL、密钥、敏感上下文。
- 开发模式下的存储降级逻辑不能在生产环境静默生效。
- 数据库里的历史脏数据不能只靠“新代码以后不再写入”解决，必要时要补数据修复脚本。
- 对安全边界关键字段，优先使用：
  - `NOT NULL`
  - 唯一约束
  - 外键约束
  - 枚举约束

## 7. 配置安全

- 必填环境变量缺失时必须失败退出，不能默认继续运行。
- 生产环境必须禁止弱占位密钥。
- 所有开发回退逻辑都要评估是否可能被误带到生产。
- 示例应用和辅助代码中不得保留默认 client secret。
- `.env.example` 必须与真实代码读取字段一致，不能残留旧命名。
- 配置文档必须按系统边界分组，例如：
  - 主服务
  - 管理后台
  - 示例应用
  - 资源服务
- 对每个变量都要写清楚：
  - 谁读取它
  - 它影响什么
  - 是否必须与数据库配置保持一致
- seed 脚本、演示脚本、安全验证脚本同样禁止默认：
  - tenant
  - issuer / base URL
  - secret
  - 端口

## 8. 运行与发布准备

- 至少要有以下安全测试：
  - 暴力破解防护
  - 账户锁定行为
  - Token 重放处理
  - Scope 提权防护
  - 租户隔离
  - CSRF 拒绝路径
- 在 CI 中运行依赖漏洞扫描，并确保使用支持 audit API 的 registry。
- 保留攻击式验证脚本，例如登录暴力破解和 Token 端点滥用脚本。
- 发布前必须有拦截机制，阻止关键密钥未配置的版本上线。
- `/ready` 必须表达真实 readiness 语义：
  - 关键依赖故障时返回 `503`
  - 不能只在 body 里写 `degraded` 却继续返回 `200`
- 健康检查不能把安全关键依赖异常伪装成“系统基本可用”。
- 攻击验证脚本也要做安全审查，不能因为“只是内部脚本”就保留弱默认值。

## 9. 资源服务与委派链路

- 目标资源校验 Token 时，不能只验 `audience`，还要强绑定：
  - `target_resource`
  - `sub`
  - `agent_id` / `app_id`
- `sub = agent:xxx` 时必须要求 `agent_id = xxx`
- `sub = app:xxx` 时必须要求 `app_id = xxx`
- 如果系统设计了 Runtime / Credential / Agent 三层关系，必须检查：
  - 无 runtime 时是否被放宽匹配
  - runtime 绑定 credential 是否可能被脱离 runtime 使用
  - target token 是否绑定到具体 target resource
- 任何缓存复用逻辑都必须把授权边界一起带进 cache key，例如：
  - principal
  - runtime
  - credential
  - connection
  - grant
  - target resource
  - external context

## 10. 常见危险信号

- “这个回退逻辑只是给开发环境用的。”
- “这个头是负载均衡器加的，所以可以直接信。”
- “这个 scope 只会从管理员角色里来，不会有问题。”
- “这是内部 ID，返回出去也没关系。”
- “登录失败审计以后再补。”
- “只做客户端级限流就够了。”
- “旧的明文私钥路径先保留兼容，后面再删。”
- “示例应用 / seed / 验证脚本不用这么严格。”
- “health check 先返回 200，后面再细化。”
- “这个默认值只是方便本地开发，不会进生产。”

如果评审中出现以上说法，应立即暂停并重新核对威胁模型，而不是直接接受。
